Пять принципов ООП от Роберта Мартина: Single Responsibility (один класс - одна причина для изменения), Open/Closed (
открыт для расширения, закрыт для модификации), Liskov Substitution (подтипы взаимозаменяемы), Interface Segregation (
много узких интерфейсов лучше одного широкого), Dependency Inversion (зависимость от абстракций, не от конкретики).
Фундамент гибкой архитектуры, предотвращает гниение кода.

## Различия между схожими принципами

### SRP vs ISP (оба про "разделение")

SRP - о классе/модуле: одна причина для изменения, одна роль в системе. ISP - об интерфейсе: клиент видит только нужные
ему методы. User может иметь одну ответственность (SRP), но выставлять разные интерфейсы для разных клиентов (ISP):
IAuthenticatable для login, IProfileEditable для UI. SRP про организацию кода, ISP про договор с клиентом. Нарушение SRP
создаёт God Object, нарушение ISP - Fat Interface.

### OCP vs DIP (оба про абстракции)

OCP - цель: расширение без модификации, достигается через полиморфизм и абстракции. DIP - механизм: зависимость от
абстракций, инверсия направления зависимостей. DIP - инструмент для достижения OCP, но не единственный (есть ещё
plugins, events). OCP говорит "что" (расширяемость), DIP - "как" (через интерфейсы). Можно соблюдать OCP через Strategy
без DIP, если конкретные стратегии создаются внутри.

### LSP vs Полиморфизм в OCP

OCP разрешает добавлять новые подтипы для расширения функциональности. LSP ограничивает, какие подтипы допустимы - они
должны соблюдать контракт базового типа. OCP про возможность расширения, LSP про корректность расширения. Можно нарушить
LSP, добавляя новый тип для OCP - например, ReadOnlyList наследует List, но бросает исключения на Add(). OCP - про
архитектуру расширяемости, LSP - про семантическую корректность иерархий.

### SRP vs High Cohesion (GRASP)

High Cohesion - методы класса тесно связаны и работают над общей концепцией. SRP - класс имеет одну причину меняться,
одного стейкхолдера. High Cohesion - техническая мера связности кода внутри. SRP - организационная мера внешних причин
изменений. Класс может иметь высокую cohesion, но нарушать SRP (User с методами изменения профиля и отправки email -
cohesive, но две ответственности). SRP строже и чётче, High Cohesion - более расплывчатая метрика.

### ISP vs DIP

ISP - клиент не должен зависеть от неиспользуемых методов интерфейса. DIP - клиент должен зависеть от абстракции, а не
конкретики. ISP про размер и содержимое интерфейса (разделяй толстые), DIP про направление зависимости (инвертируй к
абстракции). Можно соблюдать DIP с толстыми интерфейсами (нарушение ISP). Можно иметь узкие интерфейсы, но зависеть от
конкретных классов (нарушение DIP). ISP про форму контракта, DIP про природу зависимости.

### OCP vs YAGNI (конфликт)

OCP требует проектировать для расширяемости через абстракции и точки расширения. YAGNI запрещает добавлять
функциональность "на будущее". Создание абстракций "для возможности расширения потом" нарушает YAGNI. Разрешение:
применяй OCP только к точкам изменчивости, которые реально происходят (Protected Variations из GRASP). Не спекулируй
расширяемостью - добавляй абстракции при втором случае изменения, не при первом. OCP реактивный, не проактивный.
